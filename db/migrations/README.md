# Migrations

Generated by `drizzle-kit generate`. Apply with `pnpm db:migrate` to fresh databases.

## First-time setup on a NEW Postgres database

If the database is empty, just run `pnpm db:migrate`. Drizzle will apply every
migration in order.

## Existing database that was synced via `db:push`

This is our current production state — the schema was pushed before any
migration files existed. If you want to switch to migration-based workflow
on an existing database, you must mark the existing migrations as already
applied (baseline) so Drizzle doesn't try to `CREATE TABLE` rows that
already exist.

```bash
# 1. Generate the baseline (if not already in the repo)
pnpm db:generate

# 2. Mark all current migrations as applied without running them
DRIZZLE_BASELINE=true pnpm db:migrate --baseline
```

After this, future `pnpm db:migrate` runs apply only NEW migrations on top.

## Dev workflow (Sam)

For day-to-day dev, keep using `pnpm db:push` — it diffs the schema and
applies changes directly without writing a migration file. Generate a
migration file (`pnpm db:generate`) only when shipping schema changes to
production, so the same diff is applied via `pnpm db:migrate` in CI/CD.

## Why we have a single big 0000_*.sql baseline

The first migration was generated *after* significant schema work was
already in the live database. Rather than reconstruct an artificial
sequence of migrations, the baseline captures the full schema at that
point. All subsequent migrations are normal incremental diffs.
